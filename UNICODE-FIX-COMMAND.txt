━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

THE BUG:
The estimated value display shows literal "\u2013" text
instead of the actual – (en dash) character.

Example: "$473\u2013$788 Est. value" should be "$473–$788"

ROOT CAUSE:
Line 906 of ShippingCenterClient.tsx:
  ${item.valuationLow}\u2013${item.valuationHigh}

In JSX, the \u2013 escape sequence inside curly brace
expressions renders as the literal string "\u2013" instead
of the – character. It needs to use either:
  — The {"\u2013"} JSX expression pattern
  — Or the actual – character directly

The rest of the app uses {"\u2013"} pattern correctly.
This line missed it.

PART B — FIX ALL UNICODE RENDERING
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

File: app/shipping/ShippingCenterClient.tsx

For EVERY broken unicode escape found in Part A:

Change from (broken — renders as literal text):
  ${item.valuationLow}\u2013${item.valuationHigh}

Change to (correct — renders as actual character):
  ${item.valuationLow}{"\u2013"}${item.valuationHigh}

OR even simpler — use the actual character:
  ${item.valuationLow}–${item.valuationHigh}

The {"\uXXXX"} pattern is what the rest of the app uses
(see all the "\u2013", "\u00B7", "\u00D7" throughout).
Be consistent with the existing pattern.

Check these specific unicode characters:
  \u2013 = – (en dash, used for ranges like $100–$200)
  \u2014 = — (em dash)
  \u00B7 = · (middle dot, used as separator)
  \u00D7 = × (multiplication sign, used for dimensions)

ANY that appear OUTSIDE of {""} expressions in JSX
template literals will render as literal text.
